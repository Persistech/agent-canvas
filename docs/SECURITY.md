# Security & threat model

This document describes the security posture of the agent-canvas frontend
and the agent-servers it talks to. It is the canonical reference for
contributors changing security-relevant code (auth flows, header policies,
key storage, sandboxing) and for users deciding whether the default setup
meets their risk tolerance.

If you find a vulnerability, please follow our [security policy][security.md]
on GitHub rather than filing a public issue.

[security.md]: https://github.com/OpenHands/agent-canvas/blob/main/SECURITY.md

## Trust boundary

agent-canvas is a **local-machine tool**. The default deployment runs:

- the agent-server (`openhands-agent-server`) on `localhost:18000`
- the static frontend on `localhost:3001` (Vite dev) or `localhost:8000`
  (Docker all-in-one / published binary)
- an automation backend on `localhost:18001`

The browser is assumed to be running on the same machine as the agent-server,
operated by the same human. There is no multi-tenant SaaS layer between
them. The API keys the frontend stores grant access to **one** agent-server
on **your** machine, with the **same privileges you have locally**.

This is the same trust model as the AWS CLI, kubectl, Docker, or any SSH
client: the credentials are machine-local, not user-account-bound. Treat
them accordingly.

## What the keys can do

A valid session API key against the local agent-server grants the holder
the ability to:

- start / pause / resume conversations
- run shell commands inside the agent-server sandbox
- read the `PersistedSettings` blob, including LLM API keys, git
  provider tokens, and any custom secrets the user has saved
- upload / download workspace files
- read conversation history and any data the agent has touched

It does **not** grant any access beyond the agent-server itself (no other
user's data, no other host's resources). Compromised keys are bounded by
"someone can drive this agent-server on your behalf until you revoke
the key."

## How keys reach the browser

There are three paths, depending on deployment mode:

### 1. Default — generated by the launcher, never touches `localStorage`

The launcher (`npm run dev`, `npx @openhands/agent-canvas`, Docker)
generates a random 64-character session key and either:

- injects it into the Vite dev server as `VITE_SESSION_API_KEY`
  (compiled into the bundle), or
- injects it into the static-server at runtime via an inline `<script>`
  that sets `window.__AGENT_CANVAS_SESSION_API_KEY__`

In both paths the key never reaches `localStorage`. It lives in the
browser's JS heap until the tab closes. **This is the recommended
deployment** and the threat model below assumes it.

### 2. Public mode — `--public` flag

`--public` requires the user to provide a key (`LOCAL_BACKEND_API_KEY`).
The launcher does not bake the key into the bundle; instead the frontend
shows the API-key entry screen on first load and the user pastes the
key. The pasted key is persisted to `localStorage["openhands-backends"]`.

### 3. User-added custom backends

Any backend the user adds through the "Manage backends" UI (with a
non-bundled host, e.g. a remote agent-server) has its `apiKey` persisted
to `localStorage["openhands-backends"]`.

## localStorage exposure assessment

**`localStorage` is the same threat class as `~/.aws/credentials`,
`~/.kube/config`, or `~/.ssh/id_rsa`.** On the practical threat-axis:

| Attacker | Can read `localStorage`? | Practical impact |
| --- | --- | --- |
| Compromise of the user account on the machine (file read) | Yes | Same as reading any other config file |
| Malicious browser extension | Yes | Same — extensions have full DOM access anyway |
| XSS in the page | Reads `localStorage` *and* can use the key in-flight | In-flight abuse does not need `localStorage`; `localStorage` only matters for off-machine reuse |
| Disk forensics | Yes | Same as any plaintext credential on disk |
| Network observer | No | Key is never sent in cleartext on the wire |

For a thorough answer to "how bad is `localStorage`?", see the
[Security FAQ](#faq) below.

## XSS is the real attack surface

The dominant threat is **XSS via the npm dependency graph**: a compromised
or malicious transitive dependency runs JavaScript in the page context and
can do anything the page can do. Storage location is *secondary*; if XSS
happens, the attacker can drive the agent-server directly via `fetch` with
the headers the page itself uses.

Mitigations in priority order:

1. **Supply-chain hygiene.** `npm ci` for reproducible installs, minimal
   dep tree, periodic `npm audit`, Dependabot enabled. We rely on the
   `@openhands/typescript-client` package being a normal npm release
   (not a git dep) so it cannot be silently swapped.
2. **Content-Security-Policy** (see [Security headers](#security-headers)).
   The CSP shipped in this repo blocks third-party script injection and
   limits what a compromised inline script can exfiltrate to.
3. **Trusted Types** (planned; tracked below). Once every entry-point that
   builds DOM strings uses Trusted Types, we can flip on
   `require-trusted-types-for 'script'` and XSS sinks become unreachable.
4. **Backend enforcement of an Origin allow-list.** The agent-server
   already does header-only auth on `/api/*` (see
   [APIKeyHeader in the OpenAPI spec][openapi-security]) so a CSRF
   token-stealing XSS can't reuse the auth via a `<form>` POST.

[openapi-security]: https://github.com/OpenHands/agent-canvas

## Security headers

The frontend ships a Content-Security-Policy and a Permissions-Policy,
applied at two layers:

### Layer 1 — `<meta>` tag in `src/root.tsx`

`<meta httpEquiv="Content-Security-Policy" content={...}>` in the
document `<head>`. Built dynamically from the active backend registry
(`readStoredBackends()` at render time) so the `connect-src` list always
reflects the agent-servers the user has registered. Works in both Vite
dev mode and production builds without needing server cooperation.

### Layer 2 — HTTP headers from `scripts/static-server.mjs`

Defense-in-depth: the same policy is also sent as a real `Content-Security-Policy`
HTTP response header on HTML responses. HTTP headers are honored even when
a `<meta>` tag is missing (e.g. error pages, dynamically injected HTML).

CSP is omitted from `.js` / `.css` / `.svg` responses to save bytes (CSP
only governs document rendering). The broadly-applicable headers
(`X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`,
`Strict-Transport-Security`) go on every response.

The policy is intentionally permissive in `script-src` because the runtime
needs `'unsafe-inline'` for two reasons:

- React Router emits inline `<script>` tags to replay the SSR stream
  context (`window.__reactRouterContext.streamController.enqueue(...)`).
- The static-server injects an inline `<script>` to seed
  `window.__AGENT_CANVAS_SESSION_API_KEY__` etc. at runtime.

Tightening `script-src` to a nonce-based policy is **planned** and
tracked below.

### CSP directives

| Directive | Value | Why |
| --- | --- | --- |
| `default-src` | `'self'` | Default-deny: every fetch type restricted unless overridden |
| `script-src` | `'self' 'unsafe-inline' 'wasm-unsafe-eval'` + PostHog | xterm.js needs wasm-eval; PostHog loads its own JS; inline needed (see above) |
| `style-src` | `'self' 'unsafe-inline'` | Tailwind / HeroUI inject styles; Monaco injects editor styles |
| `img-src` | `'self' data: blob: https:` + every backend origin | Inline previews, avatar URLs, workspace `<img>` embeds |
| `connect-src` | `'self'` + every backend origin + PostHog + `z.openhands.dev` + `ws:` / `wss:` | Backend WebSocket / REST calls, telemetry |
| `frame-src` | `'self'` + every backend origin | Workspace artifacts embedded as `<iframe>` |
| `frame-ancestors` | `'none'` | Don't let other sites embed us |
| `base-uri` | `'self'` | Prevent `<base>` tag injection |
| `form-action` | `'self'` | Prevent exfiltration via injected forms |
| `object-src` | `'none'` | No Flash / Java / `<embed>` |
| `upgrade-insecure-requests` | (no value) | Upgrade any stray `http:` to `https:` |

## Permissions-Policy

We disable every browser feature the app does not need:

```
camera=(), microphone=(), geolocation=(), payment=(), usb=(), serial=(),
bluetooth=(), magnetometer=(), gyroscope=(), accelerometer=(),
ambient-light-sensor=(), autoplay=(), encrypted-media=(),
picture-in-picture=(), publickey-credentials-get=(self),
xr-spatial-tracking=()
```

Each disabled feature reduces the attack surface for XSS payloads and
hostile iframe embeds. WebAuthn (`publickey-credentials-get=(self)`) is
explicitly allowed for the same origin so a future passkey login flow
would not need a policy change.

## Sharing agent-canvas across users

The default deployment is single-user. If you must share the host with
other users (a lab machine, a shared dev VM, a CI runner):

- **Use `--public` and per-user API keys.** The local agent-server
  already supports multiple `OH_SESSION_API_KEYS_*` env vars. Configure
  one key per user and rotate on offboarding.
- **Never run on a multi-user machine without TLS termination in front.**
  Without HTTPS the session key is observable to anyone on the same
  network segment during the brief window it lives in the browser.
- **Run agent-server in a container per user** if you need filesystem
  isolation. The Docker all-in-one image makes this easy:
  `docker run --user <uid>:<gid> ...`.

## Hardening checklist for self-hosters

When deploying agent-canvas somewhere other than your laptop:

- [ ] **Terminate TLS in front.** Even though the local agent-server
      runs HTTP, put it behind nginx / Caddy / a cloud LB with HTTPS.
      The browser will then upgrade WS to `wss://` and the Strict-Transport-Security
      header becomes meaningful.
- [ ] **Bind the agent-server to `127.0.0.1`** if no remote users should
      reach it. Default `0.0.0.0` is fine behind a TLS-terminating proxy
      but exposes it on the LAN otherwise.
- [ ] **Set a unique `OH_SECRET_KEY`** for each deployment. The default
      auto-generated key in `~/.openhands/agent-canvas/secret-key.txt`
      is per-host — copy it across containers if you run more than one.
- [ ] **Pin `VITE_RUNTIME_SERVICES_INFO` URLs to trusted origins.**
      This string is forwarded to the agent's system prompt; an attacker
      who can write to the build environment can persuade the agent to
      `curl` arbitrary hosts.
- [ ] **Disable telemetry** if you don't want analytics events leaving
      the host: `VITE_DO_NOT_TRACK=1` and unset `VITE_POSTHOG_CLIENT_KEY`.
- [ ] **Run `npm audit --omit=dev` regularly.** We're already on `npm
      ci` for reproducible installs; complement that with periodic audits.

## FAQ

### Is `localStorage` for API keys safe?

It's **safe to the same degree as `~/.aws/credentials` or `~/.kube/config`**
— i.e., safe under the threat model "your machine and your user account
are trusted." It is not safe against:

- a malicious browser extension (which has full DOM access anyway)
- an XSS in the page (which can drive the API directly without reading
  `localStorage`)
- a local-machine compromise by malware running as you (which can read
  any file your user owns)

Moving the key to an HttpOnly cookie narrows exactly one specific attack:
"attacker exfiltrates the key string and reuses it from their own
machine." It does not narrow any of the threats above.

### Could you use a service worker to isolate the key?

No. Service workers run in the same origin as the page, can be commanded
by page JS via `postMessage`, and can be fetched and inspected by any
script in the page. They provide no isolation primitive. The only
JavaScript isolation primitive browsers ship is COOP/COEP (cross-origin
isolation), which is for Spectre mitigations, not credential isolation.

### Why not use tab-scoped storage?

There is no per-tab storage primitive in any browser. All storage
mechanisms (`localStorage`, `sessionStorage`, IndexedDB, cookies) are
**origin-scoped**, not tab-scoped. `sessionStorage` is the closest
analog — it's per-tab-per-origin — but it survives across reloads in the
same tab, doesn't persist across tab close, and is fully readable by any
script in the page.

### Why header-only auth on the agent-server API?

Cookies + the existing API surface = CSRF risk. Cookies are auto-attached
on cross-origin `<form>` POSTs and image loads, which would let a
malicious page make state-changing requests against the agent-server
under the user's session. The agent-server deliberately accepts only
header-based auth on `/api/*` (see the `session_api_keys` FastAPI
dependency in `openhands/agent_server/dependencies.py`).

The exception is the `/api/conversations/{id}/workspace*` routes, which
*do* accept cookies (`oh_workspace_session_key`). This is scoped narrowly
to file content delivery because those endpoints are reachable from
`<iframe src>` / `<img src>` where the browser cannot attach custom
headers. The cookie is HttpOnly, `SameSite=none; Secure; Partitioned`,
and tied to a single conversation. See
`src/hooks/query/use-workspace-session.ts` for the browser side.

## Planned hardening

Tracked for future PRs:

- [ ] **Nonce-based `script-src`.** Requires React Router to support
      nonce replay AND the static-server to generate a per-request nonce.
      Would let us drop `'unsafe-inline'` and add `'strict-dynamic'`.
- [ ] **Trusted Types enforcement.**
      `require-trusted-types-for 'script'` + a tiny policy that wraps
      every existing `dangerouslySetInnerHTML` / `new Function` call site.
      Currently only one `dangerouslySetInnerHTML` site exists (the
      animation keyframes in `text-shimmer.tsx`); posthog-js and
      react-syntax-highlighter will need to be audited.
- [ ] **CSP report-uri / report-to.** Stand up an endpoint (or a public
      report-only service) that aggregates CSP violation reports so we
      can detect XSS attempts in the wild.
- [ ] **Subresource Integrity for vendor chunks.** Currently we trust
      the npm supply chain end-to-end. SRI on the hashed chunk filenames
      Vite emits would catch a CDN compromise.
- [ ] **`Vary: Cookie` on the agent-server.** If the deployment ever
      serves through a CDN, the backend responses need to vary by
      session-cookie so users don't see each other's cached responses.
